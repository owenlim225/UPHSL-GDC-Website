# Dashboard Use Cases

Real-world scenarios demonstrating how various user roles interact across the UPHSL GDC Platform, highlighting data-flows and approval mechanics.

## Use Case 1: Member Game Jam Submission

**Actor:** GDC Official Member (Student)  
**Scenario:** Submitting a newly finished Unity project post-Game Jam.  

### Flow & Actions
1. The student navigates to `/login` and authenticates using their school email via Supabase Auth.
2. They are securely routed to the `/dashboard`.
3. They fill out a "New Project" form involving `Next.js Server Actions`, providing a Title, Engine (Unity), Genre, and Cover Image.
4. The user clicks **Submit**. 
5. The form data is safely pushed to Supabase under their UUID, setting `status: 'Pending'`. The Cover Image is securely pushed to the `public-assets` object bucket via a presigned-URL operation.

### Outcome
- The user sees the game sitting in their personal Dashboard timeline under "Awaiting Verification".
- The game is explicitly hidden from the public-facing `<GamesGallery />` component.

## Use Case 2: Officer Verification Pipeline

**Actor:** Technical Officer / Event Coordinator  
**Scenario:** Handling a backlog of new game submissions.

### Flow & Actions
1. Over in Discord, the webhook channel fires: `🔔 New Game Submitted: Neon Dash by Student 02-12345`.
2. The Officer clicks the link embedded in the ping and logs into the platform.
3. Their JWT token contains the explicit `{ "role": "officer" }` claim, bypassing the restrictive RLS query policies, thus allowing them to view ALL pending games across all students.
4. They navigate to `/dashboard/admin/queue`.
5. They review "Neon Dash", play-testing it locally or verifying its integrity. They click **Approve** on the platform.
6. A `PATCH` request hits the Next.js API route updating that Row's column to `status: 'Published'`.

### Outcome
- The game instantly goes live across the SSR public-facing website.
- The original student receives a dashboard notification marking success.

## Use Case 3: Public Portfolio Browsing

**Actor:** Prospective Student or Studio Recruiter (Unauthenticated)  
**Scenario:** Viewing the high-quality games made by the club.

### Flow & Actions
1. Recruiter navigates to `https://uphsl-gdc.com/projects`.
2. The page is statically generated utilizing Incremental Static Regeneration (ISR). The Next.js server frequently queries Supabase for `status: 'Published'` items and bakes the responses into the server cache.
3. The recruiter seamlessly filters games by "Genre" and "Engine" instantaneously across Edge caching.

### Outcome
- Sub-50ms page loads because unauthenticated users only interact with heavily cached Vercel Edge networks.
- Supabase traffic limits are preserved by restricting database hits entirely behind server ISR layers.
