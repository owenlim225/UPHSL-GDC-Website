# FAQ

Frequently asked questions regarding the UPHSL Game Developers' Club (GDC) platform and its underlying architecture.

## General Organization

### What is the purpose of this platform?

The platform serves dual purposes:
1. **Public Face:** A public portfolio to showcase the club, browse past game events, and view featured student games.
2. **Internal Portal:** A dedicated Dashboard where members can submit their semester projects or game jam titles directly to Officers for review and publishing.

### Who is this documentation for?

This repository documentation is written strictly for the **Engineering Team** maintaining the platform. For general student help (such as "How do I upload a massive 2GB Unity build?"), refer to the in-app Member Dashboard guidelines.

## Development & Features

### Where do uploaded games go?

When a standard member hits "Submit Project", the metadata goes to the `games` table in PostgreSQL. The actual images and/or executable `.zip` artifacts are streamed via Next.js directly into **Supabase Object Storage**, heavily locked behind Row-Level Security.

### Do I need a special account to work on this code?

Yes. To effectively work on backend database features, you must be invited to the club's "Supabase Organization" by the President. For frontend rendering UI changes, you can simply run against the staging keys without full database modification rights.

### How are Officers notified of new submissions?

We have implemented a Postgres-triggered webhook. Any time a new record is generated with a `status` of `pending`, a payload is instantly fired to the **Officer-only Discord Channel** notifying leadership to jump into the dashboard and review.

## Technical Maintenance

### What tech stack are we using?

- **Frontend:** Next.js (App Router), React, Tailwind CSS
- **Backend:** Supabase (Auth, Postgres DB, Storage)
- **Deployment:** Vercel (Edge network)

### Can I skip using Tailwind?

No. The user interface leverages strict adherence to `design-system.json` utilizing Tailwind utility classes (`bg-primary-900`, `text-accent`, etc.). Arbitrary CSS overrides are strictly banned unless approved by the design lead.
